Control systems engineering loves it’s transformations, and they can be quite overwhelming to get a hold of. And sort-of complicated too. With these series of posts, I will try to simplify and streamline the theory. The math, especially.

Let’s lay out a brief overview first.

The Time-Domain

In the real world, we have continuous-time systems. These process and generate signals continuously without any interval or wait times. For example, a DC motor. Turn on the supply and it will continuously process the applied voltage and generate a continuous torque. A continuous-time system or signal is also known as an analog system or signal.

The frequency domain

The continuous-time input is a combination of multiple frequencies and so can be the output, practically speaking. The system itself, responds differently to each frequency that passes through it. To figure out this frequency profile, we take the Fourier Transform of this input signal. If it’s a periodic signal, a Fourier Series expansion is used.

We also do the same for the system, to figure out it’s frequency response (how it changes the amplitude and phase of incoming signals with different frequencies).

The complex frequency domain (s-domain)

A real world system is expressed through differential equations, made up of derivatives with respect to the continuous-time variable, $t$. Differential equations are hard to solve. We pass these through the Laplace Transform to convert them into algebraic equations (of a complex variable - $s$). These equations in the so-called $s$-domain, are easier to solve due to their algebraic nature.

We plot the solutions to the differential equations in the time-domain - on a plane where the x-axis represents $t$. We can see how the system output changes over time, and that’s it. Kind of like watching grass grow. We can’t see through the system, to get insights about it’s control-related behaviour (not conveniently, at least).

After taking the Laplace Transform of the differential equations, we enter the $s$-domain, and here we just plot the poles and zeros of the system, but we get a lot more information about the system behavior and stability. We can visualise the changes we need to make to the input and the system to get a desirable output.

We do all of this on our notebooks[^1] and feel smart about it, but then, to implement our designed control system, we would very clearly need a computer - a micro-controller, which is a digital system. We could always use pneumatic actuators, bimetallic strips, etc. to make a mechanical control system. Actually that would be quite cool. But even this easier digital implementation is hard as it is, at least for me. I digress.

The discrete-time domain

Digital systems cannot deal with continuous values. A computer reads an instruction and executes it, according to a predefined instruction set architecture[^2]. Then it reads another and then another and it continues to operate this way. It’s a system that inherently operates sequentially.

And so, will read it’s inputs, i.e. sensor values[^3], sequentially. It’ll accept one, process it, and then take the next one, (which might already be stored in a storage buffer). There’s an interval, a gap between the reception and procession of individual sensor values or “samples”. If the real continuous-time signal will change during that interval, that change will be lost.

A digital system cannot exist in a state where it is just absorbing the external sensor values, in continuous time[^3]. Best it can do is try to take these sensor values very quickly, with very little interval between them so it can construct the continuous-time signal as closely as possible, but there will always be missed information. This process of reading inputs at fixed time intervals is called sampling.

We sample a continuous-time signal and get a discrete-time signal. Continuous-time signals are represented through differential equations, and discrete-time signals are represented through difference equations. These difference equations, are in the discrete time-domain - it is the set of all integers $\Z$, or any of it’s countable subset. On the other hand, a continuous-time domain is the set of all real numbers $\reals$, or some uncountable (infinite) subset of it1.

The solution to a difference equation is also not all that interesting, that grass all over again. So we came up with a discrete-time equivalent of the Laplace Transform - the$\text{Z}-$Transform. It helps us analyse the system stability, transient behaviour and design controllers with more insight, in the discrete-time domain.

The digital domain

There’s yet another layer to unwrap. A discrete-time signal is just that, discrete in time. But it’s amplitude is continuous - it can take values from an infinite set of values.

When we sample the amplitudes of a discrete-time signal, we get a digital signal. It’s not called sampling however, it’s called quantization - a set of discrete amplitude levels is defined[^5], and the continuous amplitude values are rounded off to the nearest value available in that set of discrete amplitude values.

A digital signal is discrete in both time and amplitude[^6]. A $z$-transform can still very well be applied to it, as digital signals are technically a subset of all discrete-time signals. Digital signal processing (DSP) utilizes $z$-transforms a lot.

Another thing I would like to add is, a discrete-time signal or a digital signal is just a large sequence (or list) of numbers. A list of paired numbers, like key-value pairs, where each time-step (sample) has an amplitude (level) associated with it. You could very well store a discrete-time signal in an excel sheet.

And, that’s it! We now have a certain high-level idea of what is being used where. Now we shall delve deeply into each of this topics.
